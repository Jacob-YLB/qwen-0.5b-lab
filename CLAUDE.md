# Claude 项目全局约束规则

本项目是一个 AI 学习型工程项目，目标不是只完成代码，而是通过代码、文档、实验记录和踩坑总结，系统学习大模型推理、微调、部署与工程化相关知识。

---

## 1. Python 代码规范

1. Python 代码遵循 PEP8 规范，统一使用 4 空格缩进。
2. 函数、类、模块命名要清晰，避免无意义命名，如 `a`、`tmp`、`test1`。
3. 重要函数需要添加简短注释，说明输入、输出和核心逻辑。
4. 对学习价值较高的代码，需要在注释中解释“为什么这么写”，而不是只解释“做了什么”。
5. 尽量避免把所有逻辑写在一个文件中，按照功能拆分模块。

---

## 2. Git 相关规则

1. 操作 Git 前先检查 `.gitignore`，不得提交 `.venv`、`__pycache__`、`*.log`、缓存文件、模型权重文件等无关内容。
2. 如果项目新增日志目录、缓存目录、模型下载目录等，需要同步更新 `.gitignore`。
3. Commit 信息简洁清晰，使用以下前缀：

   * `feat`: 新增功能
   * `fix`: 修复问题
   * `docs`: 修改文档
   * `refactor`: 重构代码
   * `test`: 添加或修改测试
   * `chore`: 配置、依赖、脚本等杂项修改
4. 每次提交尽量只包含一个明确主题，避免一次 commit 混入太多无关修改。

---

## 3. 学习内容沉淀

这是学习项目和普通工程项目最大的区别：每完成一个功能，都要尽量留下学习记录。

1. 使用 `docs/` 或 `notes/` 目录保存概念笔记，例如：

   * LoRA / QLoRA 原理
   * 为什么 QLoRA 要量化基座模型
   * KV Cache 的作用
   * vLLM 的 PagedAttention
   * Continuous Batching
   * Chat Template 与 Special Tokens
   * Tokenizer、EOS、BOS、PAD 的区别
   * 推理中的 Prefill / Decode 阶段

2. 使用 `TROUBLESHOOTING.md` 或 `FAQ.md` 记录踩坑问题，例如：

   * CUDA / PyTorch 版本不匹配
   * 显存不足 OOM
   * Hugging Face 下载超时
   * 使用 `HF_ENDPOINT=https://hf-mirror.com` 镜像
   * vLLM 的 `max-model-len` 与显存关系
   * 模型加载慢、推理慢、端口冲突、依赖冲突等问题

3. 对易错点、关键参数和注意事项，要优先写进代码注释和学习文档中。

---

## 4. AI 助手工作方式

1. 不要只给最终代码，要优先解释实现思路。
2. 对关键代码，需要说明：

   * 这个模块的作用是什么
   * 为什么要这样设计
   * 有哪些替代方案
   * 当前写法适合学习项目还是生产项目
3. 如果生成代码涉及新概念，需要补充简短的概念解释。
4. 如果代码存在风险或简化处理，需要明确标注，例如：

   * “这里是学习版写法”
   * “生产环境需要增加鉴权”
   * “这里暂未处理并发”
   * “这里暂未处理异常重试”
5. 不要过度封装。学习阶段优先保证逻辑清晰、容易理解。

---

## 5. 项目结构规则

推荐项目结构如下：

```text
project-name/
├── README.md
├── CLAUDE.md
├── requirements.txt
├── .gitignore
├── docs/
│   ├── lora_notes.md
│   ├── qlora_notes.md
│   ├── inference_notes.md
│   └── deployment_notes.md
├── notes/
│   └── daily_learning_log.md
├── src/
│   └── main.py
├── scripts/
│   └── run_demo.sh
├── tests/
│   └── test_basic.py
└── TROUBLESHOOTING.md
```

如果项目规模较小，可以简化结构，但必须保留：

* `README.md`
* `CLAUDE.md`
* `.gitignore`
* `docs/` 或 `notes/`
* `TROUBLESHOOTING.md`

---

## 6. README 写作规则

README 不只是项目说明，也要体现学习价值。

README 至少包含：

1. 项目目标
2. 技术栈
3. 项目结构
4. 环境安装
5. 运行方式
6. 当前已完成功能
7. 学习笔记索引
8. 常见问题
9. 后续计划

README 中不要只写“本项目实现了某某功能”，还要写清楚“我通过这个项目学习了什么”。

---

## 7. 实验记录规则

如果涉及模型推理、微调、部署、压测，需要记录实验结果。

建议使用 `experiments/` 目录，例如：

```text
experiments/
├── inference_test_001.md
├── finetune_lora_001.md
└── vllm_test_001.md
```

每次实验记录至少包含：

1. 实验目标
2. 使用模型
3. 硬件环境
4. 核心参数
5. 运行命令
6. 结果现象
7. 遇到的问题
8. 总结与下一步改进

---

## 8. 安全与敏感信息规则

1. 不要把 API Key、Token、账号密码写入代码。
2. 使用 `.env` 保存本地环境变量，并将 `.env` 加入 `.gitignore`。
3. 提供 `.env.example` 作为示例配置文件。
4. 不要提交模型权重、大型数据集、缓存目录。
5. 如果代码中出现密钥或隐私信息，需要提醒用户删除并更换密钥。

---

## 9. 学习优先级

本项目优先级如下：

1. 先跑通最小 Demo。
2. 再补充注释和学习文档。
3. 再优化代码结构。
4. 再考虑工程化、性能和扩展性。
5. 不要一开始就追求复杂架构。

---

## 10. 回答风格要求

当我向 AI 助手提问时，请按以下方式回答：

1. 先给结论。
2. 再解释原因。
3. 再给可执行步骤。
4. 如果涉及代码，代码后要解释关键逻辑。
5. 如果有更适合初学者的简化方案，请优先推荐。
6. 如果我的理解有偏差，请直接指出，但要说明为什么。
